# 答辩演示脚本（智能编程学习助手）

## 1. 演示目标
- 证明系统具备稳定的会话连续性与反馈持久化。
- 展示 Graph-RAG 能提供“文档/节点/关系”三类依据。
- 展示知识图谱的容错恢复和增量展开可感知交互。
- 展示工程质量门禁（type-check、前端测试、后端测试）。

## 2. 演示前准备（1 分钟）
- 启动前后端与数据库服务。
- 登录一个测试账号，确保有历史会话与知识图谱数据。
- 打开浏览器开发者工具网络面板（可选，用于展示 SSE 事件）。

## 3. 主流程演示（建议 6-8 分钟）
### 场景 A：会话连续性 + 反馈持久化
1. 在“智能问答”中新建会话，输入一个技术问题并发送。
2. 说明系统流式返回时先拿到会话元信息（`meta`），会话 ID 在首包回填。
3. 助手回答后，点击“点赞”。
4. 立即刷新页面，再进入同一会话，展示点赞状态仍正确。
5. 再次点击点赞（取消）或切换为点踩，刷新后再次确认状态一致。

话术要点：
- “现在前端不会再用最新会话猜测，而是以服务端回传的 `conversationId` 为唯一事实来源。”
- “反馈只对真实 assistant 消息 ID 生效，刷新后由消息接口统一回填。”

### 场景 B：Graph-RAG 依据可追溯
1. 在问答区提一个和图谱强相关的问题（例如 Spring Boot 与 MyBatis 的关系）。
2. 回答下方展开“依据展示”区域。
3. 依次指出三类依据：
   - 文档依据
   - 节点依据
   - 关系依据（关系类型已中文化）
4. 说明每条依据都可用于解释“为什么回答是这样”。

话术要点：
- “这不是黑盒回答，能够明确看到文档、图谱节点和关系证据。”

### 场景 C：知识图谱容错 + 增量展开
1. 进入“知识图谱”，点击任一节点打开右侧详情。
2. 点击“继续展开关联节点”，观察：
   - 新增节点/关系数量提示
   - 当前展开层级变化
   - 新增节点短暂高亮
3. 触发异常场景（可通过临时断开 Neo4j 或请求错误节点）：
   - 404 时展示“节点不存在/已移除”
   - 500 时展示“服务暂时不可用 + 重试按钮”
4. 点击“重试”展示可恢复，不出现空白页。

## 4. 质量门禁展示（1-2 分钟）
在终端依次展示：
```bash
cd frontend
npm run type-check
npm run test

cd ../backend
mvn -q test
```
说明 CI 也按同样顺序执行：
- Frontend type-check
- Frontend tests
- Backend tests

## 5. 常见评委追问与答法
- 问：刷新后点赞为什么能保持？
  - 答：反馈写入后端 `feedback` 表，消息查询接口按 `messageId + userId` 回填 `feedbackRating`，前端不依赖本地临时状态。

- 问：怎么保证新会话不会串到别的会话？
  - 答：新会话创建后，后端在 SSE `meta` 事件首包返回 `conversationId`，前端全程用该 ID 绑定消息和后续拉取。

- 问：图谱挂了会怎样？
  - 答：前端按 404/500 分级处理并提供重试，不会白屏；用户可以恢复或回退到整体图谱视图。

- 问：你怎么证明改动可靠？
  - 答：关键链路已有自动化测试覆盖，并接入 CI 门禁；本地与流水线均需通过才可合并。

- 当前系统已具备可演示、可回归、可继续扩展的工程基础。
