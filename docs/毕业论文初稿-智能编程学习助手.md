# 摘要

随着大语言模型、检索增强生成和知识图谱技术的发展，编程学习场景正在从传统的资料检索逐步转向智能化、结构化和个性化的学习辅助。传统编程学习方式普遍存在知识点分散、学习路径不清晰、问题反馈不及时、代码示例难以沉淀等问题。学习者通常需要在教程、问答社区、官方文档和本地代码之间频繁切换，学习过程割裂，难以形成稳定的知识体系。大语言模型具备较强的自然语言理解和生成能力，但在垂直知识问答中仍存在幻觉、知识滞后和回答依据不足等问题。为解决上述问题，本文设计并实现了一套基于大语言模型和知识图谱的智能编程学习助手系统。

本文首先分析了智能编程学习、检索增强生成和知识图谱技术的研究现状，并结合编程学习场景提出了系统总体设计方案。系统采用前后端分离架构，前端基于 Vue 3、TypeScript、Vite、Element Plus、ECharts 和 Monaco Editor 实现交互界面，后端基于 Spring Boot、Spring Security、MyBatis-Plus、WebFlux 和 JWT 实现业务服务。数据层采用 MySQL、Neo4j 和 Redis 的组合方式，分别用于业务数据持久化、知识图谱存储和缓存加速。

在核心功能实现方面，系统构建了融合知识文档和知识图谱的 Graph-RAG 问答流程。后端通过查询改写、文档检索、图谱节点检索和关系检索获取外部知识依据，并将文档证据、节点证据和关系证据注入大语言模型提示词中，以提升回答的准确性和可解释性。系统还实现了知识图谱导航、个性化学习路径、代码片段管理、学习进度分析、PDF 报告导出、用户注册登录和头像上传等功能。测试结果表明，系统能够较好满足编程学习场景下的智能问答、知识组织、学习记录和结果反馈需求，具有一定的工程应用价值和后续扩展空间。

关键词：智能编程学习；大语言模型；检索增强生成；知识图谱；Spring Boot；Vue 3

# Abstract

With the development of large language models, retrieval-augmented generation and knowledge graph technologies, programming learning is gradually shifting from traditional information retrieval to intelligent, structured and personalized learning assistance. Conventional programming learning methods often suffer from scattered knowledge points, unclear learning paths, delayed feedback and poor reuse of code examples. Learners usually need to switch among tutorials, Q&A communities, official documents and local code, which makes the learning process fragmented and difficult to track. Although large language models have strong natural language understanding and generation capabilities, they still face hallucination, outdated knowledge and insufficient evidence in vertical-domain question answering. To address these problems, this paper designs and implements an intelligent programming learning assistant system based on large language models and knowledge graphs.

This paper first analyzes the research status of intelligent programming learning, retrieval-augmented generation and knowledge graph technologies, and then proposes an overall system design for the programming learning scenario. The system adopts a front-end and back-end separated architecture. The front end is implemented with Vue 3, TypeScript, Vite, Element Plus, ECharts and Monaco Editor, while the back end is built with Spring Boot, Spring Security, MyBatis-Plus, WebFlux and JWT. MySQL, Neo4j and Redis are used for business data persistence, knowledge graph storage and cache acceleration respectively.

In terms of core implementation, the system constructs a Graph-RAG question-answering process that combines knowledge documents and knowledge graphs. The back end retrieves external evidence through query rewriting, document retrieval, graph node retrieval and relation retrieval, and injects document evidence, node evidence and relation evidence into the prompt of the large language model to improve the accuracy and interpretability of answers. The system also implements knowledge graph navigation, personalized learning path generation, code snippet management, learning progress analysis, PDF report export, user registration and login, and avatar upload. The testing results show that the system can meet the requirements of intelligent question answering, knowledge organization, learning tracking and feedback in programming learning scenarios, and has certain engineering application value and potential for further extension.

Keywords: intelligent programming learning; large language model; retrieval-augmented generation; knowledge graph; Spring Boot; Vue 3

# 第 1 章 引言

## 1.1 研究背景与意义

随着软件技术的快速发展，编程能力已经成为计算机、物联网工程、人工智能和相关交叉学科人才培养中的重要能力。编程学习不仅要求学习者掌握语言语法，还要求理解框架原理、工程规范、数据库设计、接口调试、部署运维等多方面知识。对于初学者而言，知识点分散、概念关联复杂、实践反馈周期长是学习过程中常见的问题；对于具备一定基础的进阶学习者而言，如何高效组织知识、复用代码经验、定位薄弱环节和形成持续学习路径，也是影响学习效率的重要因素。

传统编程学习方式主要依赖教材、课程视频、技术博客、官方文档和问答社区。这些资源虽然数量丰富，但存在几个明显不足。第一，知识组织方式往往以章节、文章或单个问题为单位，知识点之间的依赖关系不够清晰，学习者容易形成碎片化理解。第二，传统搜索引擎返回的是网页列表，需要学习者自行筛选、比较和整合，检索成本较高。第三，编程学习强调实践，但代码示例、调试经验和高频模板常常散落在不同位置，难以沉淀为可复用资产。第四，学习过程缺乏持续记录和反馈，学习者很难客观了解自己的学习轨迹、知识覆盖情况和近期薄弱点。

近年来，大语言模型在自然语言理解、代码生成和知识问答方面表现出较强能力。以 ChatGPT、DeepSeek、Qwen、ChatGLM 等模型为代表的新一代大模型能够根据用户自然语言输入生成较完整的解释、步骤和代码示例，为编程学习提供了新的交互方式。然而，大语言模型并不能完全替代可靠的知识来源。在实际应用中，大模型可能产生看似合理但实际错误的回答，也可能无法获取用户本地知识库、课程资料和项目上下文中的最新内容。这种现象通常被称为模型幻觉或知识滞后问题。对于编程学习而言，错误的配置项、过时的 API 用法和缺少依据的解释都可能误导学习者。

检索增强生成技术为解决上述问题提供了有效思路。该技术在模型生成回答之前，先从外部知识库中检索与用户问题相关的资料，再将检索结果作为上下文提供给大语言模型，使模型回答能够参考明确的知识来源。与单纯依赖模型参数知识相比，检索增强生成能够提升回答的时效性、领域适配性和可追溯性。与此同时，知识图谱能够以节点和关系的形式表达知识之间的依赖、包含和关联关系，适合用于组织编程知识体系。例如，Spring Boot 与 Spring MVC、Spring Security、MyBatis、MySQL、Redis 等知识点之间存在明显的学习先后关系和工程组合关系，仅靠文本检索难以直观呈现，而知识图谱可以帮助学习者从结构上理解这些关系。

因此，将大语言模型、检索增强生成和知识图谱引入编程学习场景，具有较强的研究意义和应用价值。本文设计并实现的智能编程学习助手，不仅提供智能问答功能，还将知识图谱导航、个性化学习路径、代码片段管理和学习进度分析整合到同一平台中，形成“提问、理解、沉淀、跟踪、复盘”的学习闭环。该系统的意义主要体现在以下几个方面。

第一，从学习支持角度看，系统能够降低学习者查找资料和整合知识的成本，使用户通过自然语言提问获得结构化回答，并通过来源信息了解回答依据。第二，从知识组织角度看，系统利用知识图谱展示编程知识点之间的关系，帮助用户从单点问题扩展到系统化学习。第三，从工程实践角度看，项目综合使用前后端分离、JWT 认证、流式响应、图数据库、缓存、RAG 检索和容器化部署等技术，具有较好的工程训练价值。第四，从应用推广角度看，该系统可作为课程辅学平台、企业培训系统或个人知识管理工具的基础原型，具有进一步扩展的可能。

## 1.2 国内外研究现状及发展趋势分析

### 1.2.1 智能问答与智能编程学习系统研究现状

智能问答系统是自然语言处理和信息检索领域的重要研究方向。早期问答系统多基于规则、模板和关键词匹配，能够在限定场景下回答固定类型问题，但对复杂语义、上下文关系和开放域问题的适应能力较弱。随着深度学习的发展，基于神经网络的问答模型逐渐提升了语义理解能力，可以从文档中抽取答案或生成自然语言回答。近年来，大语言模型的出现进一步推动问答系统从“检索匹配”向“理解生成”转变，使系统能够根据用户问题生成更加连贯、自然和结构化的回答。

在编程学习领域，问答系统常见于在线教育平台、编程社区和代码辅助工具中。国外许多平台已经围绕代码解释、错误诊断、学习路径推荐和自动反馈开展探索。大语言模型在代码生成、代码补全、错误解释、单元测试生成等任务中表现突出，使编程学习者能够通过对话方式获得即时帮助。国内智能教育平台也开始引入大模型问答、题目推荐和学习分析能力，用于提高学习资源匹配效率和用户学习体验。

然而，现有智能编程学习系统仍存在不足。一方面，许多系统主要关注问答本身，缺少对知识结构和学习过程的持续管理。用户虽然可以获得一次性答案，但难以将回答转化为长期可追踪的学习路径。另一方面，部分系统依赖通用大模型直接回答问题，缺少对本地知识库、课程资料和用户历史行为的结合，回答的可解释性和个性化程度不足。此外，代码片段、学习记录、知识图谱等学习资产在许多系统中相互割裂，尚未形成完整闭环。

本文所设计的智能编程学习助手尝试解决这些问题。系统将智能问答作为入口，将知识图谱作为结构化知识支撑，将学习路径和学习记录作为过程管理手段，将代码片段作为经验沉淀工具，使用户不只是获得即时答案，还能够形成持续积累的学习资源。

### 1.2.2 检索增强生成与知识图谱技术研究现状

检索增强生成技术最早受到开放域问答和知识密集型自然语言处理任务的推动。Lewis 等提出的 RAG 模型将外部文档检索与生成模型结合起来，使模型在生成答案时能够参考检索到的文档内容。此后，相关研究逐步扩展到查询改写、文档分块、向量检索、混合检索、重排序、上下文压缩和答案引用等多个环节。RAG 技术的发展趋势表明，单纯依赖向量相似度检索已经难以满足复杂应用需求，结合关键词检索、结构化数据、知识图谱和用户反馈的混合检索方案正在成为重要方向。

知识图谱是一种用节点和边表示实体及其关系的知识组织方式。与普通文本不同，知识图谱能够明确表达概念之间的依赖、包含、关联等关系，因此适合用于学习路径推荐、知识导航和关系推理。在编程学习场景中，知识图谱可以表示 Java、Spring Boot、MyBatis、MySQL、Redis、Docker 等知识点之间的先修关系和工程组合关系。学习者通过图谱可以更直观地理解“为什么要先学某个知识点”以及“某个框架与其他技术如何组合使用”。

近年来，Graph-RAG 的思想进一步将知识图谱与检索增强生成结合起来。与普通 RAG 只检索文本片段不同，Graph-RAG 会同时利用图谱节点、实体关系和社区结构等信息，为生成模型提供更加结构化的上下文。对于编程学习而言，这种方式可以使回答不仅包含文档内容，还能说明知识点之间的关系。例如，当用户询问“Spring Boot 和 MyBatis 是什么关系”时，系统可以同时检索 Spring Boot、MyBatis、MySQL 等节点及其关系，从而给出更具解释性的回答。

总体来看，国内外研究均表明，大语言模型、RAG 和知识图谱结合具有较高应用潜力。但在具体工程落地中，仍需要解决文档质量、检索粒度、关系召回、生成约束、系统性能和用户体验等问题。本文结合毕业设计项目实现了一个面向编程学习场景的 Graph-RAG 应用原型，对上述技术进行了工程化整合。

## 1.3 研究内容与目标

本文的研究内容围绕智能编程学习助手系统的设计与实现展开，主要包括以下几个方面。

第一，分析编程学习场景的实际需求，确定系统功能边界。系统面向编程初学者、课程学习者和具备一定基础的进阶用户，提供智能问答、知识图谱、学习路径、代码片段和学习进度分析等功能。

第二，设计基于大语言模型和知识图谱的 Graph-RAG 问答流程。系统通过查询改写扩展用户问题关键词，通过知识文档检索获取文本证据，通过 Neo4j 图谱检索获取节点和关系证据，并将这些证据组织为提示词上下文，再调用大语言模型生成回答。

第三，完成前后端分离系统的工程实现。后端使用 Spring Boot 构建 REST API 与 SSE 流式接口，使用 Spring Security 和 JWT 实现认证授权，使用 MyBatis-Plus 操作 MySQL，使用 Neo4j Java Driver 访问图数据库，使用 Redis 缓存问答结果和上下文。前端使用 Vue 3、TypeScript、Element Plus、ECharts 和 Monaco Editor 构建主要页面。

第四，设计并实现学习辅助闭环。系统通过学习路径模块帮助用户规划学习顺序，通过代码片段模块沉淀常用模板，通过学习进度模块记录用户行为并生成可视化报告，使学习过程能够被持续追踪。

第五，对系统进行功能测试和基本验证。通过注册登录、智能问答、图谱浏览、路径生成、片段管理、进度统计和报告导出等测试用例验证系统功能完整性，并结合自动化测试提升系统可靠性。

本文的总体目标是实现一个功能较完整、结构清晰、可运行演示且具备扩展能力的智能编程学习助手系统，为大语言模型和知识图谱技术在编程学习场景中的应用提供工程实践参考。

# 第 2 章 相关理论与技术

## 2.1 大语言模型

大语言模型是基于大规模语料训练的深度神经网络模型，能够对自然语言文本进行理解、生成、摘要、翻译、推理和代码生成等任务。Transformer 架构提出后，自注意力机制显著提升了模型对长文本上下文和复杂语义关系的建模能力，为后续大规模预训练语言模型的发展奠定了基础。随着参数规模、训练语料和指令微调方法的发展，大语言模型逐渐具备了较强的通用任务能力，并被广泛应用于智能问答、内容生成、代码辅助、教育辅导和知识管理等场景。

在编程学习场景中，大语言模型具有明显优势。首先，模型可以理解用户自然语言问题，将模糊描述转换为较清晰的问题意图。其次，模型可以生成结构化解释、实现步骤和代码示例，帮助学习者降低理解门槛。再次，大语言模型支持多轮对话，可以结合历史上下文继续回答追问，适合用于学习过程中的连续交流。最后，大模型可以对代码错误、配置问题和概念差异进行解释，提升学习反馈效率。

但是，大语言模型也存在局限。模型回答依赖训练数据和提示词上下文，当问题涉及最新框架版本、本地项目资料或特定课程内容时，模型可能无法准确回答。模型还可能生成看似合理但实际错误的内容，尤其在 API 名称、配置项、依赖版本和边界条件方面需要谨慎。因此，在面向编程学习的系统中，不能只依赖大模型本身，还需要结合外部知识检索、来源展示、提示词约束和用户反馈机制。

本系统通过统一的 LlmProvider 接口封装不同模型提供方，使系统能够接入 OpenAI、DeepSeek、Qwen、ChatGLM 等兼容服务。后端在 ChatService 中组织系统提示词和对话上下文，并通过流式接口将模型输出实时返回给前端。该设计既保证了模型接入灵活性，也便于后续根据成本、延迟和回答质量切换模型。

## 2.2 检索增强生成技术

检索增强生成是一种将外部知识检索与生成模型结合的技术路线。其基本思想是在生成回答之前，先根据用户问题从知识库中检索相关内容，再将检索结果作为上下文输入给大语言模型，使模型回答建立在可追溯资料基础上。与单纯依赖模型参数记忆相比，RAG 可以提升回答的时效性、专业性和可信度。

一个完整的 RAG 流程通常包括知识索引、信息检索和回答生成三个阶段。

### 2.2.1 知识索引

知识索引阶段的目标是将原始知识资料处理成便于检索的结构。原始资料可以是 Markdown 文档、网页、PDF、数据库记录或人工整理的知识片段。系统通常需要对文本进行清洗、切分、向量化和存储。文本切分是影响检索效果的重要环节。如果切分粒度过大，检索结果中可能包含大量无关内容；如果切分粒度过小，片段可能缺少必要上下文，导致模型难以理解完整含义。因此，系统需要在片段长度和上下文完整性之间取得平衡。

本项目在知识文档模块中将知识文档保存到 MySQL 的 knowledge_document 表，并进一步生成知识分块记录。DocumentService 中设置文本分块大小为 500，重叠长度为 120，使相邻片段之间保留一定上下文连接。系统还为文档和分块生成本地向量表示，并在可选情况下写入 Elasticsearch 向量索引，从而支持后续语义检索。

### 2.2.2 信息检索

信息检索阶段的目标是从知识库中找出与用户问题最相关的内容。传统关键词检索依赖词项匹配，适合处理明确术语和关键词；向量检索则将文本映射到向量空间，通过相似度计算获取语义相关内容。两种方式各有优缺点。关键词检索可解释性较强，但对同义表达和语义改写不够敏感；向量检索能够处理语义相似问题，但可能忽略关键术语或产生不稳定召回。

本系统采用混合检索方式。首先，QueryRewriteService 对用户问题进行关键词提取和同义词扩展，例如将“Spring Boot”扩展为“springboot”“spring-boot”“starter”，将“缓存”扩展为“cache”“缓存击穿”“缓存穿透”等相关词。其次，DocumentService 同时执行语义检索和关键词检索，并对结果进行融合排序。最后，KnowledgeRetrievalService 进一步从 Neo4j 中检索相关知识节点和节点关系，为回答生成提供图谱证据。

### 2.2.3 回答生成

回答生成阶段是将用户问题、历史对话和检索证据整合后交给大语言模型生成回答。本阶段的关键在于提示词设计。提示词需要明确系统角色、回答语言、回答结构、知识优先级和禁止编造原则。若检索到的资料不足，模型应说明哪些内容是推断，而不能伪造来源。

本系统的系统提示词要求模型优先使用检索到的知识上下文，结合图谱关系证据进行解释，并在可能情况下按照结论、解释、实现步骤、示例代码、常见问题和参考依据的结构回答。该设计使回答更贴近编程学习需求，也便于前端展示来源信息和用户反馈。

## 2.3 知识图谱与 Graph-RAG

知识图谱是一种结构化知识表示方式，通常由实体节点、关系边和属性组成。与普通文本文档相比，知识图谱能够明确表示概念之间的关系。在编程学习场景中，知识图谱可以将 Java、Spring Boot、Spring MVC、Spring Security、MyBatis、MySQL、Redis、Docker 等知识点组织为有向关系网络。例如，Spring Boot 可能包含 Spring MVC，MyBatis 与 MySQL 具有相关关系，Spring Security 与 JWT 共同支撑认证授权流程。

Neo4j 是常用的图数据库，使用节点和关系存储数据，并通过 Cypher 查询语言检索图结构。本项目使用 Neo4j 存储编程知识点及其关系，知识节点包含 id、name、category、description、difficulty、keywords 等属性，关系类型主要包括 DEPENDS_ON、CONTAINS 和 RELATED_TO。GraphService 提供图谱概览、节点详情、邻接节点展开、搜索和健康检查等能力。

Graph-RAG 是在普通 RAG 基础上引入图谱证据的增强方法。普通 RAG 主要检索文档片段，而 Graph-RAG 还会检索相关实体和关系，使回答具备更强的结构化解释能力。本系统的 KnowledgeRetrievalService 会同时获取文档证据、图谱节点证据和图谱关系证据，并将其统一构造成模型上下文。这样，当用户提出与知识点关系相关的问题时，系统可以不仅回答概念本身，还能解释知识点之间的先后、包含和关联关系。

## 2.4 系统开发相关技术

本系统采用前后端分离架构，涉及多种工程技术。

后端使用 Spring Boot 作为基础框架。Spring Boot 通过自动配置机制降低了 Java Web 应用开发复杂度，适合快速构建 REST API、业务服务和配置管理。本项目使用 Spring Boot Web 提供 HTTP 接口，使用 Spring Boot Validation 完成请求参数校验，使用 WebFlux 和 Reactor Flux 实现 SSE 流式问答输出。

系统认证授权基于 Spring Security 和 JWT 实现。用户登录成功后，后端生成 JWT 令牌，前端在后续请求中携带令牌访问受保护接口。后端通过安全过滤器解析令牌并识别用户身份，从而实现用户数据隔离。密码存储采用加密方式，避免明文保存。

数据访问层使用 MyBatis-Plus。MyBatis-Plus 在 MyBatis 基础上提供通用 CRUD、条件构造器和分页支持，能够减少重复数据访问代码。本项目通过 mapper、entity、dto 和 vo 分层组织数据结构，使接口输入输出与数据库实体保持清晰边界。

前端使用 Vue 3、TypeScript、Vite、Element Plus、Pinia、Vue Router、ECharts 和 Monaco Editor。Vue 3 提供响应式组件模型，TypeScript 提升大型前端项目可维护性，Vite 提供快速构建能力，Element Plus 提供常用 UI 组件，Pinia 负责状态管理，Vue Router 负责页面路由，ECharts 用于知识图谱和学习图表展示，Monaco Editor 用于代码片段编辑与展示。

数据层使用 MySQL、Neo4j 和 Redis。MySQL 保存用户、会话、消息、反馈、学习路径、代码片段和知识文档等结构化数据；Neo4j 保存知识节点和关系；Redis 用于缓存问答结果和上下文，提高重复问答场景下的响应效率。

部署层采用 Docker Compose 和 Nginx。Docker Compose 能够统一编排 MySQL、Neo4j、后端服务、前端构建和 Nginx 反向代理，降低部署环境差异。Nginx 用于托管前端静态资源并反向代理后端接口。

# 第 3 章 智能编程学习助手的 Graph-RAG 架构

## 3.1 知识文档处理与文本分块

智能编程学习助手需要回答大量与编程语言、框架、数据库、部署工具和工程实践相关的问题。为了使回答能够参考本地知识资料，系统设计了知识文档管理模块。知识文档既可以来自官方文档摘要，也可以来自课程资料、学习笔记和人工整理的知识条目。每条知识文档包含标题、正文、分类和创建时间等信息，并存储在 MySQL 的 knowledge_document 表中。

仅保存整篇文档并不能满足精确检索需求。若文档过长，系统检索到的内容可能包含大量无关段落，模型生成回答时难以判断重点；若仅根据标题检索，则可能遗漏正文中的关键内容。因此，系统在新增知识文档时，会对文档内容进行分块处理。当前实现中，DocumentService 将分块大小设置为 500，重叠长度设置为 120。该设置的目的在于使每个分块保持较完整语义，同时通过重叠部分避免相邻段落被硬切分后丢失上下文。

文档处理流程如下：用户新增知识文档后，系统首先保存原始文档记录，然后更新本地语料统计信息，生成整篇文档向量。接着，系统对文档内容进行分块，为每个分块生成向量表示，并保存到 knowledge_chunk 表中。如果启用了 Elasticsearch 向量索引，系统还会将文档向量和分块向量写入外部向量索引，以便后续进行更高效的向量检索。若未启用外部索引，系统则使用本地 TF-IDF 向量和余弦相似度完成检索。

这种设计兼顾了系统可运行性和后续扩展性。一方面，本地 TF-IDF 向量不依赖外部嵌入模型，便于在毕业设计环境中稳定运行；另一方面，系统预留了 ElasticsearchVectorService，可在后续替换为更强的向量数据库或专业嵌入模型。

## 3.2 基于 TF-IDF 的本地向量表示

向量表示是语义检索的重要基础。对于给定文本，系统需要将其转换为固定维度向量，再通过余弦相似度衡量问题与文档之间的相关程度。本项目当前实现了轻量级本地 TF-IDF 向量服务 EmbeddingService。该服务不依赖外部模型，能够在本地完成文本分词、词频统计、逆文档频率计算和向量归一化。

EmbeddingService 设置向量维度为 512。系统首先使用正则表达式提取英文单词和中文词项，并过滤长度过短的 token。然后统计文本中的词频，根据语料库中的文档频率计算 TF-IDF 权重。由于词表规模不固定，系统使用哈希桶方式将不同词项映射到固定维度向量中。最后，对向量进行 L2 归一化，便于后续计算余弦相似度。

TF-IDF 权重计算公式如下：

TF-IDF(t, d) = TF(t, d) × IDF(t)

其中，TF(t, d) 表示词项 t 在文档 d 中出现的频率，IDF(t) 表示词项 t 的逆文档频率。逆文档频率可表示为：

IDF(t) = log((N + 1) / (DF(t) + 1)) + 1

其中，N 表示语料库文档总数，DF(t) 表示包含词项 t 的文档数量。系统在启动时会加载已有知识文档，初始化文档频率统计；新增文档时会更新语料统计信息。

为了提升编程领域常见词项的匹配效果，系统内置了 spring、boot、redis、mysql、java、python、vue、docker、api、jwt、security、mybatis、transaction、nginx、maven、git、algorithm 等常见编程词项。这样即使在初始语料较少时，系统也能对常见技术关键词形成基本表示。

与深度语义嵌入模型相比，TF-IDF 本地向量的语义表达能力有限，尤其难以处理复杂同义改写和长距离语义关系。但在毕业设计应用场景中，该方案具有实现简单、运行稳定、成本较低、可解释性较强等优点。系统通过查询改写、关键词检索和图谱检索对其进行补充，以提升整体检索效果。

## 3.3 基于查询改写和混合排序的信息检索

### 3.3.1 检索算法设计

用户提问通常具有口语化和不完整性。例如，用户可能输入“mp 怎么分页”“redis 序列化咋配”“SpringBoot 登录怎么做”等问题。如果系统直接使用原始问题检索，可能遗漏标准技术名词和相关概念。因此，本系统在检索前引入查询改写机制。

QueryRewriteService 首先对用户问题进行归一化处理，将文本转换为小写，并提取英文、数字、下划线、连字符和中文词项。然后，系统根据内置同义词表扩展关键词。例如：

1. “spring boot” 可扩展为 “springboot”“spring-boot”“starter”。
2. “redis” 可扩展为 “redis缓存”“redis cache”“缓存”。
3. “mybatis-plus” 可扩展为 “mybatis plus”“mp”“分页插件”。
4. “vue3” 可扩展为 “vue 3”“composition api”“script setup”。
5. “jwt” 可扩展为 “token”“鉴权”“登录态”。

扩展后的关键词会用于文档关键词检索和图谱节点检索。DocumentService 的检索流程由语义检索、关键词检索和融合排序组成。语义检索部分会先尝试分块级检索，根据用户问题向量与知识分块向量计算相似度；如果启用了 Elasticsearch，则优先使用外部向量索引；如果没有可用索引，则回退到本地向量计算。关键词检索部分会根据扩展词项匹配文档标题、分类和内容。最后，系统将两类结果合并，并根据语义分数、关键词分数和排名奖励进行排序，选出 TopK 文档作为文档证据。

KnowledgeRetrievalService 在文档检索之外，还会使用扩展关键词检索 Neo4j 知识图谱。系统在 Knowledge 节点的 name、category 和 keywords 属性中查找匹配项，最多召回若干相关节点。若召回节点之间存在关系，系统进一步检索这些节点之间的 DEPENDS_ON、CONTAINS、RELATED_TO 等关系，形成图谱关系证据。

最终，系统的检索结果由四部分组成：文档证据、图谱节点证据、图谱关系证据和扩展关键词。这些结果被统一封装为 RetrievalResult，并交给回答生成阶段使用。

### 3.3.2 评价指标

为了评估检索效果，可从准确性、召回能力、排序质量和响应性能四个方面设计指标。

第一，准确率 Precision@K。该指标表示前 K 个检索结果中相关结果所占比例。对于编程学习问答，Precision@K 越高，说明返回给模型的上下文越集中，越不容易引入干扰信息。

第二，召回率 Recall@K。该指标表示所有相关资料中被前 K 个结果覆盖的比例。对于知识文档较多的场景，Recall@K 能反映系统是否容易漏掉关键资料。

第三，命中率 Hit@K。该指标表示前 K 个结果中是否至少存在一个相关结果，适合用于评估问答系统是否能找到可用依据。

第四，平均倒数排名 MRR。该指标关注第一个相关结果出现的位置，适合评估排序算法是否能够将最有用资料放在前面。

第五，响应时间。对于交互式学习系统，检索耗时会直接影响用户体验。系统应记录检索总耗时、语义检索命中数、关键词检索命中数和最终命中数，以便后续优化。

当前项目中 RetrievalMetricsService 已记录检索摘要信息，包括语义命中数、关键词命中数、最终命中数和耗时。后续可进一步构建人工标注问答集，对 Precision@K、Recall@K、Hit@K 和 MRR 进行量化评估。

### 3.3.3 检索效果验证方案

由于毕业设计项目的知识文档规模相对有限，本文采用功能验证与日志指标结合的方式进行检索效果验证。验证问题可覆盖 Spring Boot、Redis、MyBatis-Plus、Vue 3、MySQL、Docker、JWT 等常见知识主题。每个问题人工指定期望命中文档或图谱节点，系统执行检索后检查 TopK 结果是否包含相关资料。

表 3.1 为检索验证用例示例。

## 3.4 基于知识图谱证据的问答生成

回答生成是 Graph-RAG 流程的最后阶段。系统需要将检索结果组织为大语言模型能够理解的上下文，并通过提示词约束模型回答。KnowledgeRetrievalService 的 buildContext 方法会按照固定结构拼接上下文，主要包括以下内容。

第一，文档证据。系统列出检索到的知识文档标题、分类和内容摘要，使模型能够参考具体文本资料。

第二，图谱节点证据。系统列出相关知识节点名称、分类、描述和关键词，帮助模型理解用户问题涉及哪些知识点。

第三，图谱关系证据。系统列出相关节点之间的关系，例如 “Spring Boot --RELATED_TO--> MyBatis” 或 “JWT --RELATED_TO--> Spring Security”。这些关系能够增强回答的结构化解释能力。

第四，扩展关键词。系统保留查询改写后的关键词，便于模型理解用户问题可能涉及的同义表达。

ChatService 在系统提示词中要求模型始终使用中文回答，优先使用检索知识上下文，不编造配置项、API 名称或框架行为。当图谱关系存在时，模型应结合节点和关系证据进行解释。若检索知识不足，模型应说明哪些内容属于推断。

前端通过 SSE 接收模型流式输出。用户发送问题后，后端首先创建或更新会话，保存用户消息，并返回 meta 事件告知前端 conversationId。随后，大语言模型生成的内容以 delta 事件逐步返回，输出完成后发送 done 事件并保存 assistant 消息。系统还会保存 sources 字段，用于记录回答对应的文档、节点和关系依据。用户可对回答进行有用或无用反馈，反馈数据写入 feedback 表，为后续优化检索和问答质量提供依据。

该设计使系统具备三个特点。第一，回答过程能够结合外部知识，降低模型幻觉风险。第二，来源信息能够回传前端，使回答具备可追溯性。第三，流式输出降低用户等待感，适合交互式学习场景。

## 3.5 本章小结

本章介绍了智能编程学习助手的 Graph-RAG 架构。系统通过知识文档处理、文本分块、本地 TF-IDF 向量表示、查询改写、混合检索、图谱节点与关系召回，为大语言模型提供外部知识依据。与普通问答系统相比，该架构不仅关注文本文档匹配，还引入知识图谱关系，使回答更适合编程学习中的结构化理解需求。下一章将进一步介绍系统整体设计、模块实现和测试情况。

## 4.4 系统功能实现

### 4.4.1 用户认证与个人中心模块实现

用户认证模块由 AuthController 和 AuthService 实现。系统提供注册、登录、退出、当前用户查询、个人摘要和头像上传接口。用户注册时，系统校验用户名、邮箱和密码，并使用加密算法保存密码。用户登录成功后，系统生成 JWT 令牌返回前端。前端将令牌保存到本地，并在后续请求中通过 Authorization 请求头携带令牌。

后端通过 Spring Security 配置安全过滤链，对受保护接口进行身份校验。若请求未携带有效令牌，系统返回未授权响应；若用户访问不属于自己的数据，业务层抛出异常并由全局异常处理器转换为统一响应。用户头像上传后保存在后端 uploads 目录，并在用户信息中保存访问路径。

该模块为系统所有个性化功能提供基础，保证会话、代码片段、学习路径和学习进度等数据能够按用户隔离。

### 4.4.2 智能问答模块实现

智能问答模块由 ChatController 和 ChatService 实现，是系统核心模块。用户在前端聊天页面输入问题后，前端调用 /api/chat/send 接口。该接口使用 text/event-stream 类型返回 SSE 流式响应。

ChatService 的处理流程如下。

第一，判断请求是否包含 conversationId。若没有，系统创建新会话并生成标题；若包含，则校验会话是否属于当前用户。

第二，保存用户消息到 message 表。

第三，根据问题内容生成哈希值，尝试从 Redis 缓存中获取已有答案。若缓存命中，系统直接保存 assistant 消息并通过 SSE 返回缓存内容。

第四，若缓存未命中，系统调用 KnowledgeRetrievalService 执行 Graph-RAG 检索，获取文档证据、节点证据和关系证据，并构造模型上下文。

第五，系统读取对话上下文。若 Redis 中存在压缩后的上下文，则直接使用；若不存在，则从数据库读取历史消息并进行压缩。

第六，调用 LlmProviderFactory 获取当前模型提供方，并以 Flux 形式接收模型输出。每当模型返回一个文本片段，后端立即通过 delta 事件发送给前端。

第七，模型输出完成后，系统保存 assistant 消息、缓存问答结果、更新上下文缓存，并返回 done 事件。

该模块还提供会话列表、消息列表、会话删除、回答反馈、检索指标查询和指标重置接口。用户反馈写入 feedback 表，消息查询时会根据 messageId 和 userId 回填反馈状态，使页面刷新后仍能保持点赞或点踩结果。

### 4.4.3 知识图谱模块实现

知识图谱模块由 GraphController 和 GraphService 实现。后端通过 Neo4j Driver 执行 Cypher 查询，并将图谱数据封装为 GraphVO 返回前端。图谱概览接口查询最多 500 个知识节点和 500 条关系，前端使用 ECharts 渲染为可交互图谱。

节点详情接口根据节点 id 查询 Knowledge 节点，并返回名称、分类、描述、难度和关键词等信息。邻接节点接口支持深度参数，系统将深度限制在 1 到 4 之间，避免一次性展开过多节点导致性能问题。搜索接口根据关键词匹配节点名称、分类和关键词，帮助用户快速定位知识点。关联文档接口根据图谱节点查询相关知识文档，使图谱与文档库形成连接。

GraphService 还对常见节点提供中文描述补全，并限制允许的节点分类和关系类型，保证前端展示数据更加规范。若 Neo4j 查询失败，系统会返回明确错误信息，前端可展示重试入口，避免页面空白。

### 4.4.4 学习路径模块实现

学习路径模块由 PathController 和 PathService 实现。用户输入学习目标后，系统根据知识图谱结构和模型能力生成学习路径。每条学习路径保存到 learning_path 表中，路径中的每个知识节点保存到 learning_node 表中。节点状态包括 todo、doing 和 done，用于表示未开始、进行中和已完成。

学习路径模块的设计目标是解决编程学习中“知道问题答案，但不知道下一步学什么”的问题。系统将用户目标拆解为多个有序知识节点，并为节点关联学习资源。用户可以在前端查看路径列表、切换节点状态和删除路径。学习节点状态变化也可作为学习记录的一部分，为后续进度分析提供依据。

在后续优化中，路径生成可以进一步结合用户历史问答、薄弱知识点、代码片段使用情况和图谱依赖关系，使路径更加个性化。

### 4.4.5 代码片段与学习进度模块实现

代码片段模块由 SnippetController 和 SnippetService 实现。用户可以新增代码片段，填写标题、代码内容、语言、描述和标签。系统支持分页查询、关键词搜索、标签过滤、片段推荐、更新、删除、使用次数记录、反馈、导入和导出。前端通过 Monaco Editor 或代码高亮组件展示代码，使用户可以方便地阅读和复用。

代码片段模块的意义在于将学习过程中的代码经验沉淀下来。编程学习不仅需要理解概念，也需要积累可复用模板，例如 Spring Boot 跨域配置、MyBatis-Plus 分页查询、Vue 组合式函数、Redis 分布式锁、Dockerfile 多阶段构建等。系统通过标签和搜索功能帮助用户管理这些模板。

学习进度模块由 ProgressController 和 ProgressService 实现。系统根据 learning_record 等数据统计用户学习情况，生成仪表盘、智能建议、周计划、热力图、雷达图和 PDF 报告。报告导出使用 iTextPDF 实现，能够将学习概览、趋势指标和建议内容整理为 PDF 文件。该模块使用户能够看到学习过程中的活跃程度和知识覆盖情况，从而形成持续改进的反馈闭环。

# 第 5 章 总结与展望

## 5.1 论文总结

本文围绕“基于大语言模型和知识图谱的智能编程学习助手系统设计与实现”这一课题，完成了需求分析、理论研究、架构设计、功能实现和测试验证等工作。

首先，本文分析了编程学习场景中知识点分散、学习路径不清晰、问题反馈不及时和代码经验难以沉淀等问题，说明了将大语言模型、检索增强生成和知识图谱引入编程学习辅助系统的必要性。其次，本文介绍了大语言模型、RAG、知识图谱、Graph-RAG、Spring Boot、Vue 3、MySQL、Neo4j 和 Redis 等相关理论与技术，为系统实现奠定基础。

在系统设计方面，本文提出了基于前后端分离的总体架构，将系统划分为表示层、业务服务层、AI 协调层和数据层。系统使用 Vue 3 和 Element Plus 构建前端页面，使用 Spring Boot 构建后端服务，使用 MySQL 保存业务数据，使用 Neo4j 保存知识图谱，使用 Redis 支撑缓存。系统功能包括用户认证、智能问答、知识文档、知识图谱、学习路径、代码片段和学习进度分析。

在核心技术实现方面，本文设计并实现了面向编程学习场景的 Graph-RAG 问答流程。系统通过查询改写扩展用户问题，通过文档检索获取文本证据，通过图谱检索获取知识节点和关系证据，再将这些证据注入大语言模型提示词中生成回答。系统支持 SSE 流式输出、会话历史保存、回答来源记录、用户反馈和问答缓存，提升了学习问答的交互体验和可追溯性。

在工程实现方面，系统已经形成较完整的功能闭环。用户可以注册登录、进行多轮智能问答、浏览知识图谱、生成学习路径、管理代码片段、查看学习仪表盘并导出学习报告。系统还支持 Docker Compose 部署，具备一定可维护性和扩展性。

总体而言，本文完成了一个面向编程学习场景的智能学习助手原型系统。该系统不局限于一次性问答，而是将知识组织、路径规划、代码沉淀和进度反馈结合起来，为大语言模型在教育学习场景中的工程应用提供了实践参考。

## 5.2 未来展望

虽然系统已经完成主要功能，但仍存在一些不足，需要在后续工作中进一步优化。

第一，当前系统的本地向量表示主要基于 TF-IDF，语义理解能力与专业嵌入模型相比仍有限。后续可以引入 bge、text-embedding、m3e 等中文或多语言嵌入模型，并结合向量数据库提升语义检索效果。

第二，当前查询改写和混合排序仍以规则和启发式策略为主，缺少大规模标注数据支持。后续可以构建编程学习问答评测集，使用 Precision@K、Recall@K、MRR、NDCG 等指标系统评估检索质量，并根据用户反馈持续优化排序策略。

第三，当前知识图谱数据规模相对有限，主要覆盖常见编程技术栈。后续可以扩展更多课程知识、框架版本、项目案例和资源节点，使图谱能够支持更丰富的学习路径推荐。

第四，学习路径生成还可以更加个性化。后续系统可结合用户历史问答、已完成节点、薄弱领域、代码片段使用情况和测试结果，生成更符合用户能力水平的路径。

第五，系统安全性和运维能力仍可加强。后续可以加入接口限流、日志脱敏、异常监控、模型调用超时控制、敏感词过滤和更完善的权限管理机制。

第六，前端交互体验还可以进一步优化。例如，回答来源可以更直观地展示文档、节点和关系依据；知识图谱可以支持更细致的筛选和布局；学习仪表盘可以加入趋势对比和阶段目标提醒。

未来，随着大语言模型、代码智能体和教育数据分析技术的发展，智能编程学习助手可以进一步演进为面向课程教学、实验训练、项目实践和职业能力提升的综合平台。本文所实现的系统为这一方向提供了初步工程基础。
