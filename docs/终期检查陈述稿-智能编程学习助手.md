# 终期检查陈述稿-智能编程学习助手

## 一、项目定位与建设目标
本项目名称为“智能编程学习助手”。项目面向计算机相关课程学习者，核心目标是解决三个痛点：  
1. 学习资料分散，检索效率低；  
2. 只问不练，知识掌握不扎实；  
3. 学习过程缺少可视化反馈与阶段评估。  

围绕以上问题，系统构建了“问答-图谱-路径-练测-看板”的学习闭环，覆盖从知识获取到能力检验的完整过程。

## 二、当前完成情况与新增功能
截至终期检查，系统已完成核心功能开发并可完整运行，新增功能主要如下：

1. 新增“学习看台”模块  
支持学习资源检索、推荐、收藏、观看历史；新增“上传到本地学习资源库”能力，实现本地视频文件入库管理与播放。

2. 新增“练与测”模块  
支持练习题生成、考试题生成、作答判分、结果汇总，形成“学后即练、练后可测”的闭环。

3. 完善知识图谱编辑闭环  
在原有“新增节点/新增关系”基础上，新增“删除节点/删除关系”操作，图谱维护能力从“可增加”升级为“可治理”。

## 三、系统核心功能点总结（答辩可重点陈述）

1. 智能问答（SSE 流式）  
支持多轮对话与流式输出，降低等待体感；结合知识检索减少大模型幻觉。

2. RAG 检索增强回答  
通过文档库、向量检索、图谱关联召回多源融合，为生成提供可追溯上下文。

3. 知识图谱探索与治理  
可视化展示知识节点与关系，支持查询、扩展邻居、关系创建与删除操作。

4. 学习路径规划  
根据学习目标自动生成分阶段学习节点，并支持节点状态推进。

5. 学习看台（新增）  
聚合线上与本地学习资源，支持“看-收藏-复看-继续学”。

6. 练与测（新增）  
将问答结果转化为可检验题目，支持即时反馈与阶段测评。

7. 学习看板  
输出热力图、雷达图、智能洞察与报告，帮助用户复盘学习行为。

## 四、技术架构与各技术在项目中的具体使用

### 4.1 前端：Vue3 + Vite + Element Plus

1. Vue3（Composition API）  
用于页面逻辑组织和组件化开发。项目按“视图页（views）+ 复用组件（components）+ 状态管理（Pinia stores）”分层，提升可维护性。

2. Vite  
作为构建与开发服务器，提供快速热更新，缩短联调与迭代时间；生产环境用于前端打包产物输出。

3. Element Plus  
用于表单、表格、弹窗、按钮等基础 UI 能力。项目中的登录注册、知识文档管理、图谱操作弹窗、练测交互等都基于该组件库实现，保证一致的交互规范。

### 4.2 后端：Spring Boot + Spring Security + MyBatis-Plus

1. Spring Boot  
作为后端主框架，负责 REST API 暴露、依赖注入、配置管理、模块解耦。控制层（Controller）负责接口，服务层（Service）负责业务编排。

2. Spring Security（JWT）  
实现登录认证与接口鉴权：登录后签发 JWT，后续请求携带 token；除白名单接口外，`/api/**` 统一鉴权；未登录返回 401，越权返回 403。

3. MyBatis-Plus  
负责 MySQL 访问与 ORM 映射，简化 CRUD；对复杂筛选与排序场景使用 XML 自定义 SQL（如片段筛选、路径节点查询等）。

### 4.3 数据层：MySQL + Redis + Neo4j + Elasticsearch

1. MySQL  
存储事务型结构化数据：用户、会话、消息、学习路径、练测会话、学习记录、知识文档元数据等。

2. Redis  
用于热点缓存与上下文缓存，例如问答缓存、会话上下文缓存，减少重复计算与响应延迟。

3. Neo4j  
存储知识节点与关系边，支撑知识图谱展示、邻居扩展与关系探索。

4. Elasticsearch  
用于关键词检索与向量检索（包括文档级与分块级索引），支撑 RAG 的高效召回。

### 4.4 部署：Docker Compose
使用 Docker Compose 编排 MySQL、Redis、Neo4j、Elasticsearch、Backend、Nginx、Frontend 构建容器，实现一键启动与环境一致性，便于验收和复现。

## 五、RAG（检索增强生成）详细过程

本项目的 RAG 不是“先检索再拼接”的单一步骤，而是一个多阶段流程：

### 第 1 步：用户问题接入与预处理
前端提交问题后，后端先进行基础清洗（空白处理、长度限制、上下文拼接），并可做关键词扩展，增强召回覆盖率。

### 第 2 步：Query 向量化（Embedding）
当前项目采用本地 TF-IDF 向量化实现 embedding：  
1. 文本分词并计算 TF；  
2. 结合语料统计计算 IDF；  
3. 通过哈希桶映射为固定 512 维向量；  
4. 向量归一化。  

说明：该方案属于“轻量 embedding”，优势是离线可运行、成本可控、可复现；后续可平滑替换为神经 embedding（如 BGE/OpenAI）。

### 第 3 步：多通道召回
系统并行进行三类召回：

1. 向量召回  
使用 query 向量在 Elasticsearch 向量索引中检索相似文档/知识块。

2. 关键词召回  
根据关键词在文档索引中做文本匹配，补足纯向量召回可能遗漏的精确术语。

3. 图谱关联召回  
结合 Neo4j 中的知识关系，召回与主题高度相关的邻接知识节点及关联文档。

### 第 4 步：融合、去重与重排
将多源结果按相关度、来源质量、主题一致性进行融合，删除重复内容，并按最终得分重排，得到高质量候选上下文。

### 第 5 步：上下文构建（Context Building）
把重排后的内容组织成结构化上下文，包含知识片段与来源信息，控制总长度，避免提示词超长导致生成不稳定。

### 第 6 步：提示词组装与模型生成
将“系统提示词 + 用户问题 + 检索上下文”组装后调用 LLM，采用流式返回（SSE）逐段输出答案，提升交互体验。

### 第 7 步：结果回传与可追溯性
后端返回答案的同时返回来源信息；前端可展示外部资料或文档引用，保证答案可追踪、可核验。

### 第 8 步：行为记录与持续优化
系统记录问答行为、反馈结果和召回指标，供学习看板统计，并为后续检索策略优化提供依据。

### 第 9 步：降级策略
当向量服务或图谱服务不可用时，系统可退化到关键词检索或基础问答，保证主要功能可用而非整体失败。

## 六、为何当前不直接使用外部 Embedding 模型（答辩说明）

1. 课程项目阶段优先保证可复现与可演示；  
2. 外部模型受网络、配额和成本影响较大；  
3. 本地处理有助于减少数据外发；  
4. 当前实现已模块化封装，后续可低成本升级。  

因此这是阶段性工程取舍，不是能力上限。

## 七、终期检查可展示的完整业务闭环

1. 用户登录后提问，系统基于 RAG 输出答案；  
2. 进入知识图谱查看节点关系，必要时新增/删除节点关系；  
3. 进入学习路径推进学习节点；  
4. 在学习看台查看或上传学习资源并继续学习；  
5. 在练与测中完成练习与测验并获得评分反馈；  
6. 在学习看板查看热力图、雷达图与洞察报告。  

该流程能够完整展示系统从“知识获取”到“能力评估”的全链路能力。

## 八、后续优化方向

1. 接入神经 embedding，进行 A/B 对比提升语义检索效果；  
2. 引入重排模型提升检索上下文质量；  
3. 加强练测错因分析与个性化学习建议；  
4. 完善图谱数据治理与自动清洗能力。  

## 九、结语
本项目已完成从单一问答系统到“学-练-测-评”闭环学习系统的升级。终期版本重点保证了功能完整性、可演示性与可扩展性；新增“学习看台”和“练与测”有效增强了教学场景价值，为后续智能化演进打下了工程基础。
