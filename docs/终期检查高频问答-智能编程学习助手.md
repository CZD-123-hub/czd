# 终期检查高频问答（智能编程学习助手）

## 1. 你的项目要解决的核心问题是什么，目标用户是谁？
答：项目聚焦“编程学习过程中信息分散、练习反馈慢、学习路径不清晰”三个问题。目标用户是计算机相关专业学生和初中级自学者。系统把智能问答、知识图谱、学习路径、练与测、学习看台、学习看板整合为一个闭环，帮助用户从“问问题”走到“学会并能练”。

## 2. 相比普通问答系统，你的系统创新点在哪里？
答：创新点是“学练测闭环”而不是单点问答。具体包括：RAG 检索增强回答、知识图谱可视化探索、自动学习路径、练习与测评、学习行为统计看板，以及代码片段沉淀复用。新增模块“学习看台”和“练与测”进一步强化了资源学习和能力检验链路。

## 3. 为什么前后端选 Vue3 + Spring Boot，而不是其他技术栈？
答：Vue3 + Vite 开发效率高、组件生态成熟，适合快速搭建多模块前端；Spring Boot 生态完整，适合中后台业务和多数据源整合。两者分层清晰、文档完善、上手成本可控，适合课程项目在有限时间内稳定交付。

## 4. 为什么数据层同时用 MySQL + Redis + Neo4j + Elasticsearch？各自边界是什么？
答：MySQL 存事务性结构化数据（用户、会话、练习、学习记录）；Redis 做热点缓存与会话上下文缓存；Neo4j 存知识节点与关系，支持图查询；Elasticsearch 做关键词和向量检索，支撑 RAG 召回。边界原则是“事务进 MySQL、关系网进 Neo4j、检索进 ES、高频临时态进 Redis”。

## 5. 系统整体架构是怎样分层的？请从一次请求链路讲一下
答：前端（Vue）发起请求到后端 Controller，Controller 做参数接收与鉴权，Service 编排业务流程，Mapper/Repository 访问 MySQL/Neo4j/ES/Redis，最后统一封装响应返回前端。以问答为例：前端发送问题 -> 后端检索知识与图谱上下文 -> 调用 LLM 生成 -> SSE 流式返回 -> 存储消息与行为记录。

## 6. JWT 鉴权流程是怎样的？登录后怎么做接口权限控制？
答：用户登录成功后后端签发 JWT，前端保存 token 并在请求头携带。Spring Security 过滤器校验 token 合法性和过期时间，解析用户身份写入安全上下文。除登录注册等白名单接口外，`/api/**` 统一要求认证，未认证返回 401，越权返回 403。

## 7. 你的 RAG 流程具体步骤是什么（召回、重排、拼接上下文、生成）？
答：流程是：问题预处理与关键词扩展 -> 向量召回（ES 向量索引）+ 关键词召回（文本索引）+ 图谱关联召回（Neo4j）-> 融合去重与相关性排序 -> 组装上下文 prompt -> 调用 LLM 生成答案 -> 返回答案和可追溯来源。这样既提高准确性，也降低幻觉风险。

## 8. 为什么当前用本地 TF-IDF 向量化，不直接用外部 embedding 模型？
答：这是阶段性工程取舍。第一，离线可运行、可复现，演示稳定；第二，避免外部 API 成本、配额和网络不确定性；第三，减少学习数据外发风险；第四，已将向量化模块独立封装，后续可平滑替换为 BGE 或 OpenAI embedding 模型。

## 9. TF-IDF 向量化在项目里是怎么做成固定维度向量的？
答：文本先分词并统计 TF，再结合语料统计计算 IDF，得到每个词的 TF-IDF 权重。然后通过哈希桶映射到固定 512 维向量并累加，最后做 L2 归一化。新增文档后更新语料频次，用于后续 IDF 动态调整。

## 10. 图谱模块中 Neo4j 存什么，MySQL 存什么，如何关联？
答：Neo4j 主要存知识节点、关系边及图结构属性；MySQL 存文档、学习记录、用户行为等事务数据。关联方式是通过 `knowledgeId/nodeId` 这样的业务键在两侧映射，例如文档可关联图谱节点，节点详情可反查关联文档。

## 11. 新增的“学习看台”模块核心价值是什么？
答：学习看台用于聚合学习资源，把“问答后的知识点”转化为“可继续学习的视频/资料”。它支持搜索、推荐、收藏、观看历史、在线导入和本地上传，帮助用户在同一平台完成“查-学-回看”。

## 12. 学习看台里的“本地上传资源”流程是怎样的，做了哪些校验？
答：前端点击“上传到本地资源库”选择文件，后端接收 multipart 文件并校验扩展名（如 mp4/webm/ogg/m4v）和大小上限，生成文件名保存到 `uploads/videos`，再写入 `learning_video` 表并可选自动收藏。成功后前端刷新列表并可直接播放。

## 13. 新增的“练与测”模块怎么出题、怎么判题、怎么统计结果？
答：练与测分为练习与考试两类会话。出题阶段结合用户历史薄弱点和知识主题生成题目；练习模式支持逐题作答与即时反馈，考试模式支持提交后统一评分；结果写入会话与题目记录，生成正确率、错因、趋势等统计用于学习看板。

## 14. 学习路径模块如何生成节点，节点状态如何推进？
答：用户输入学习目标后，系统结合知识库与图谱关系生成有序节点列表（由浅入深）。节点状态有 `todo/in-progress/done`，用户可逐步推进，状态变化会同步写入学习记录，后续用于看板分析与推荐。

## 15. 学习看板的指标（热力图、雷达图、洞察）数据来源是什么？
答：核心来源是 `learning_record` 行为日志与练测结果、学习路径进度等。热力图按日期聚合学习活跃度；雷达图按能力维度聚合学习行为；智能洞察对近期数据做规则分析，生成学习建议和周计划调整提示。

## 16. 代码片段推荐是基于什么策略，如何避免“推荐无关”？
答：策略是会话上下文关键词匹配 + 标签/语言过滤 + 历史使用效果加权。为避免无关推荐，会做主题约束、去重和最低相关度阈值控制，并允许用户对推荐结果做“有用/无用”反馈，反向优化后续推荐。

## 17. 知识图谱新增节点/关系后，为什么还要补删除操作？如何保证一致性？
答：没有删除操作会导致脏数据积累，图谱会逐渐失真。补齐删除节点与删除关系后，图谱维护形成完整 CRUD。实现上先做存在性校验，再执行删除，并在前端刷新图谱与健康指标，保证可见状态与存储状态一致。

## 18. 如果 Elasticsearch 或 Neo4j 临时不可用，系统如何降级？
答：设计为“可降级不崩溃”。当 ES 不可用时，问答可退化为关键词检索或有限上下文回答；当 Neo4j 不可用时，图谱相关功能受限，但文档检索、问答与学习路径基础能力仍可运行。系统通过异常捕获和统一错误返回避免整体失败。

## 19. 你做过哪些异常处理与统一错误返回设计？
答：后端通过全局异常处理器统一封装错误码、消息、路径和 traceId；鉴权错误由 Security 层统一返回 401/403；业务异常抛出可读信息，前端通过 `resolveErrorMessage` 统一提取提示文案，提升可排障性和用户体验一致性。

## 20. 项目性能瓶颈可能在哪里，做过哪些优化（缓存、索引、分页、异步）？
答：潜在瓶颈在大模型响应、向量检索、图查询和大列表渲染。已做优化包括：Redis 缓存问答上下文与热点结果；MySQL 建索引并分页查询；问答采用 SSE 流式返回降低等待体感；前端按需加载与状态复用，减少重复请求。

## 21. 数据库索引是怎么设计的？哪些字段是高频查询字段？
答：围绕高频条件建立索引，如 `user_id`、`conversation_id`、`created_at`、`document_id`、`category`、`knowledge_id` 等；联合索引用于时间维度列表和行为统计。设计目标是覆盖“按用户查、按时间查、按主题查、按关联查”四类核心查询。

## 22. 你做了哪些测试（接口测试、服务测试、前端状态测试）？覆盖哪些核心链路？
答：后端有 Controller 合约测试和 Service 单元测试，覆盖认证、问答、文档、图谱、路径、进度、片段等主链路；前端有 Pinia store 测试（如 chat、graph）。重点保障登录鉴权、核心接口契约、图谱加载与问答状态流转。

## 23. Docker Compose 部署时，服务启动顺序和健康检查怎么处理？
答：通过 `depends_on` + healthcheck 控制依赖服务先就绪（MySQL、Redis、Neo4j、ES）再启动后端；Nginx 代理前后端并提供统一入口。这样可以降低“容器启动了但服务不可用”的假成功情况。

## 24. 当前版本的局限性是什么？下一步准备怎么演进（如切换神经 embedding）？
答：局限主要是本地 TF-IDF 语义能力有限、推荐仍偏规则、自动评测解释性可继续增强。下一步计划：接入神经 embedding 做 A/B 对比、引入向量重排模型、完善图谱自动抽取与清洗、提升学习建议的个性化程度。

## 25. 如果再迭代两周，你会优先做哪三件事，为什么？
答：第一，替换或并行接入神经 embedding，显著提升语义检索质量；第二，完善练与测结果分析（错因标签、薄弱点追踪），直接提升教学价值；第三，增强图谱治理（批量校验、冲突检测、回滚），提高数据质量和系统可维护性。

## 结尾陈述（可在答辩收尾使用）
答：本项目已经从“智能问答原型”发展为“学-练-测-评”闭环学习系统。当前版本重点保证了可运行、可演示、可扩展；新增的学习看台和练与测模块补齐了学习场景深度。后续将以语义检索升级和个性化学习反馈为重点持续迭代。
