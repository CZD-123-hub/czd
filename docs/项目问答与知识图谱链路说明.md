# 项目问答与知识图谱链路说明

这份文档用于复习项目中两个核心流程：

1. 一次用户问答会经过哪些文件和方法。
2. 知识图谱是如何产生、查询并展示到前端的。

---

## 一、一次用户问答的完整链路

整体可以理解为：

```text
前端发送问题
  -> 后端接收请求
  -> 创建或校验会话
  -> 保存用户消息
  -> 检查 Redis 问答缓存
  -> 缓存未命中时执行 RAG 检索
  -> 拼接 systemPrompt
  -> 调用大模型
  -> SSE 流式返回前端
  -> 保存回答和缓存结果
```

一次问答涉及的核心文件和方法顺序可以简记为：

```text
ChatView.handleSend
-> chatStore.sendMessage
-> chatApi.sendMessageSSE
-> ChatController.sendMessage
-> ChatService.sendMessage
-> ChatService.createQuestionHash
-> ChatCacheService.getCachedAnswer
-> KnowledgeRetrievalService.retrieve
-> QueryRewriteService.expandTerms
-> DocumentService.search
-> DocumentService.searchSemantic
-> DocumentService.searchSemanticByChunk
-> ElasticsearchVectorService.searchChunkIds / searchIds
-> EmbeddingService.embed / cosineSimilarity
-> DocumentService.searchByKeywords
-> KnowledgeRetrievalService.retrieveNodesFromNeo4j
-> KnowledgeRetrievalService.retrieveRelationsFromNeo4j
-> KnowledgeRetrievalService.buildContext
-> ChatService 拼接 systemPrompt
-> ChatCacheService.getConversationContext
-> MessageMapper.selectByConversationIdOrderByCreatedAt
-> LlmProviderFactory.getProvider
-> OpenAiProvider/QwenProvider/ChatGlmProvider.chatStream
-> ChatService 返回 meta / delta / done
-> ChatCacheService.cacheAnswer
-> QueryRetrievalLogService.logRetrieved
-> 前端 chatStore 更新聊天消息
```

### 1. 前端聊天页面触发发送

文件：

```text
frontend/src/views/Chat/ChatView.vue
```

核心方法：

```ts
handleSend()
```

作用：

用户点击发送按钮或按快捷键后，`handleSend()` 会读取输入框内容，然后调用聊天 store：

```ts
chatStore.sendMessage(content)
```

这一层主要负责页面交互，不直接请求后端。

### 2. 前端 Pinia 管理聊天状态

文件：

```text
frontend/src/stores/chat.ts
```

核心方法：

```ts
sendMessage(content: string)
```

作用：

这个方法会先在前端本地插入两条临时消息：

```text
用户消息
空的 assistant 消息
```

这样页面可以立即显示用户问题和“正在回复”的占位效果。

然后调用：

```ts
chatApi.sendMessageSSE(activeConversationId, content, callbacks)
```

### 3. 前端通过 SSE 请求后端

文件：

```text
frontend/src/api/chat.ts
```

核心方法：

```ts
sendMessageSSE(conversationId, content, callbacks)
```

作用：

它使用 `fetch` 发起请求：

```text
POST /api/chat/send
Accept: text/event-stream
```

请求体包含：

```json
{
  "conversationId": 1,
  "content": "用户问题"
}
```

后端返回的是 SSE 流式数据，前端会解析这些事件：

```text
meta：会话 ID
delta：模型返回的一小段内容
done：回答完成
error：错误信息
```

### 4. 后端 ChatController 接收请求

文件：

```text
backend/src/main/java/com/coding/assistant/controller/ChatController.java
```

核心方法：

```java
sendMessage(@Valid @RequestBody ChatSendRequest request)
```

作用：

这个接口对应：

```text
POST /api/chat/send
```

它会：

```text
1. 获取当前登录用户 ID。
2. 记录一次 CHAT 学习行为。
3. 调用 ChatService.sendMessage(...)。
4. 把 ChatService 返回的事件包装成 ServerSentEvent。
```

核心代码逻辑：

```java
return chatService.sendMessage(userId, request)
        .map(event -> ServerSentEvent.<String>builder()
                .event(event.event())
                .data(event.data())
                .build());
```

### 5. ChatService 进入问答主流程

文件：

```text
backend/src/main/java/com/coding/assistant/service/ChatService.java
```

核心方法：

```java
sendMessage(Long userId, ChatSendRequest request)
```

这是一次问答的后端主流程。

它首先处理会话：

```text

```

涉及 Mapper：

```text
ConversationMapper.insert(...)
ConversationMapper.selectById(...)
ConversationMapper.updateById(...)
```

然后保存用户消息：

```text
MessageMapper.insert(userMessage)
```

```java
createQuestionHash(String content)
```

作用：

对用户问题做标准化，然后计算 MD5：

```text
trim()
toLowerCase()
UTF-8 bytes
MD5
```

得到：

```text
questionHash
```

然后调用：

```text
ChatCacheService.getCachedAnswer(questionHash)
```

### 7. Redis 问答缓存

文件：

```text
backend/src/main/java/com/coding/assistant/service/ChatCacheService.java
```

核心方法：

```java
getCachedAnswer(String questionHash)
cacheAnswer(String questionHash, String answer, String sourcesJson)
```

缓存 key：

```java
"chat:qa:" + questionHash
```

如果缓存命中：

```text
1. 不再进行 RAG 检索。
2. 不再调用大模型。
3. 直接返回缓存答案。
4. 保存 assistant 消息。
5. 记录 cache hit 日志。
```

如果缓存未命中，继续执行 RAG 检索。

### 8. RAG 检索入口

文件：

```text
backend/src/main/java/com/coding/assistant/service/KnowledgeRetrievalService.java
```

核心方法：

```java
retrieve(String question)
```

作用：

同时检索四类信息：

```text
1. 扩写关键词 expandedTerms
2. 文档证据 documents
3. 图谱节点证据 nodes
4. 图谱关系证据 edges
```

核心逻辑：

```java

```

### 9. 用户问题关键词扩写

文件：

```text
backend/src/main/java/com/coding/assistant/service/QueryRewriteService.java
```

核心方法：

```java
expandTerms(String query)
```

作用：

根据固定同义词表对用户问题扩写。

例如：

```text
knn
  -> 近邻搜索
  -> 向量检索
  -> 相似度检索

```

扩写结果会用于：

### 10. 文档检索总入口

文件：

```text
backend/src/main/java/com/coding/assistant/service/DocumentService.java
```

核心方法：

```java
search(String query, int topK)
```

作用：

根据用户问题从知识文档库中检索相关文档，用于 RAG。

它会同时做两路检索：

```java
searchSemantic(...)
searchByKeywords(...)
```

也就是：

```text
语义检索
+
关键词检索
```

最后把两路结果融合打分、排序，返回最相关的文档。

### 11. 文档语义检索

文件：

```text
backend/src/main/java/com/coding/assistant/service/DocumentService.java
```

核心方法：

```java
searchSemantic(String query, int topK)
```

作用：

先把用户问题转成向量：

```java
embeddingService.embed(query)
```

然后按下面优先级检索：

```text
1. 优先检索 chunk。
2. chunk 没命中，再检索整篇文档向量。
3. Elasticsearch 不可用或无结果时，回退到本地 embedding 余弦相似度。
```

### 12. 优先检索 chunk

文件：

```text
backend/src/main/java/com/coding/assistant/service/DocumentService.java
```

核心方法：

```java
searchSemanticByChunk(List<Double> queryVector, List<String> expandedTerms, int topK)
```

作用：

优先用问题向量去检索文档分块。

它先调用：

```java
elasticsearchVectorService.searchChunkIds(queryVector, candidateSize)
```

如果命中 chunk，会做二次重排：

```text
语义相似度 semantic
关键词命中 keyword
ES 原始排名 rankBonus
```

重排公式大概是：

```text
semantic * 0.7 + keyword * 0.3 + rankBonus * 0.2
```

最后按文档去重，只保留每篇文档最相关的 chunk。

注意：

```text
返回类型仍然是 KnowledgeDocument，
但 content 会被替换成命中的 chunk 摘要。
```

所以最终拼进 systemPrompt 的 `[Document Evidence] Excerpt` 很可能就是 chunk 内容。

### 13. Elasticsearch 向量检索

文件：

```text
backend/src/main/java/com/coding/assistant/service/ElasticsearchVectorService.java
```

核心方法：

```java
searchChunkIds(List<Double> queryVector, int topK)
searchIds(List<Double> queryVector, int topK)
searchIdsInIndex(String indexName, List<Double> queryVector, int topK)
```

作用：

在 Elasticsearch 的 `dense_vector` 字段上做 KNN 检索。

核心逻辑：

```java
.knn(knn -> knn
    .field("embedding")
    .queryVector(queryVectorFloat)
    .k(k)
    .numCandidates(candidates)
)
```

其中：

```text
searchChunkIds：查 chunk 级向量索引。
searchIds：查文档级向量索引。
```

### 14. 本地向量化和相似度计算

文件：

```text
backend/src/main/java/com/coding/assistant/service/EmbeddingService.java
```

核心方法：

```java
embed(String text)
cosineSimilarity(List<Double> a, List<Double> b)
```

作用：

`embed()` 使用本地 TF-IDF 思路把文本转成固定维度向量。

`cosineSimilarity()` 计算两个向量的余弦相似度。

如果 ES 向量检索失败，`DocumentService` 会用这里的余弦相似度做本地兜底。

### 15. 文档关键词检索

文件：

```text
backend/src/main/java/com/coding/assistant/service/DocumentService.java
```

核心方法：

```java
searchByKeywords(List<String> keywords, int topK)
scoreKeyword(KnowledgeDocument doc, List<String> keywords)
```

作用：

根据扩写关键词匹配文档：

```text
标题 title
分类 category
正文 content
```

权重：

```text
标题命中：+4
分类命中：+2
正文命中：+1
```

这一路结果会和语义检索结果融合。

### 16. 检索 Neo4j 图谱节点

文件：

```text
backend/src/main/java/com/coding/assistant/service/KnowledgeRetrievalService.java
```

核心方法：

```java
retrieveNodesFromNeo4j(List<String> expandedTerms)
```

作用：

用扩写关键词查询 Neo4j 的 `Knowledge` 节点。

匹配字段包括：

```text

```

### 17. 检索 Neo4j 图谱关系

文件：

```text
backend/src/main/java/com/coding/assistant/service/KnowledgeRetrievalService.java
```

核心方法：

```java
retrieveRelationsFromNeo4j(List<KnowledgeNodeVO> nodes)
```

作用：

只在已经召回的节点之间查关系：

```cypher
MATCH (a:Knowledge)-[r]->(b:Knowledge)
WHERE a.id IN $nodeIds AND b.id IN $nodeIds
RETURN a.id AS source, b.id AS target, type(r) AS type
```

这样可以避免把无关关系引入 prompt。

## 二、知识图谱产生、查询和展示链路

整体可以理解为：

```text
后端连接 Neo4j
  -> 启动时初始化图谱种子数据
  -> 提供图谱查询接口
  -> 前端加载 nodes + edges
  -> Pinia 保存图谱状态
  -> ECharts 渲染图谱画布
```

知识图谱产生和展示涉及的核心文件和方法顺序可以简记为：

```text
Neo4jConfig.neo4jDriver
-> Neo4jDataInitializer.initNeo4jData
-> Neo4jDataInitializer.ensureSchema
-> Neo4jDataInitializer.INIT_CYPHER 创建 Knowledge 节点和关系
-> GraphController.getOverview
-> GraphService.getOverview
-> GraphService.mapToKnowledgeNodeVO
-> GraphService.localizeDescription
-> GraphVO(nodes + edges) 返回前端
-> frontend/src/api/graph.ts getOverview
-> graphStore.loadOverview
-> KnowledgeGraphView.vue 计算 displayedNodes / displayedEdges
-> GraphRenderer.props 接收 nodes / edges
-> GraphRenderer.buildOption
-> utils/echarts.ts 注册 GraphChart
-> GraphRenderer.initChart
-> init(chartContainer.value)
-> chart.value.setOption(buildOption())
-> ECharts graph series 渲染到画布
```

用户点击或操作图谱时的补充链路：

```text
点击节点
-> GraphRenderer emit nodeClick
-> KnowledgeGraphView.handleNodeClick
-> graphStore.selectNode
-> GraphController.getNodeDetail
-> GraphService.getNodeDetail
-> 右侧详情面板展示 selectedNode

继续展开关联节点
-> graphStore.loadNeighbors
-> GraphController.getNeighbors
-> GraphService.getNeighbors
-> 合并并去重 nodes / edges
-> GraphRenderer.updateChart
-> chart.value.setOption(...)
```

### 1. 配置 Neo4j 连接

### 2. 启动时初始化图谱数据

文件：

```text
backend/src/main/java/com/coding/assistant/config/Neo4jDataInitializer.java
```

核心方法：

```java
initNeo4jData()
```

触发时机：

```java
@EventListener(ApplicationReadyEvent.class)
```

也就是 Spring Boot 启动完成后执行。

它会先创建约束和索引：

```java
ensureSchema(session)
```

包括：

```cypher
CREATE CONSTRAINT knowledge_id IF NOT EXISTS
CREATE CONSTRAINT knowledge_name IF NOT EXISTS
CREATE INDEX knowledge_category IF NOT EXISTS
CREATE INDEX knowledge_difficulty IF NOT EXISTS
```

然后检查是否已经有图谱数据：

```cypher
MATCH (n:Knowledge) RETURN count(n) AS cnt
```

如果已有节点，就跳过初始化。

如果没有节点，就执行：

```java
session.run(INIT_CYPHER)
```

### 3. INIT_CYPHER 创建节点和关系

文件：

```text
backend/src/main/java/com/coding/assistant/config/Neo4jDataInitializer.java
```

核心常量：

```java
INIT_CYPHER
```

它会创建 `Knowledge` 节点。

每个节点包含：

```text
id
name
category
difficulty
description
keywords
```

例如：

```text
Java

Vue.js
MySQL
Redis
Docker
RAG
```

同时创建关系：

```text
DEPENDS_ON：依赖关系
CONTAINS：包含关系
RELATED_TO：相关关系
```

比如：

```text
Spring Boot --DEPENDS_ON--> Java
MyBatis Plus --DEPENDS_ON--> MyBatis
Vue Router --DEPENDS_ON--> Vue.js
```

### 7. 查询单个节点详情

### 22. 图谱链路一句话总结

```text
后端通过 Neo4jConfig 创建 Neo4j Driver，
Neo4jDataInitializer 在空库时初始化 Knowledge 节点和关系，
GraphService 用 Cypher 查询节点和边并封装成 GraphVO，
GraphController 暴露 /api/graph 接口，
前端 graph.ts 请求接口并用 Pinia 保存 nodes 和 edges，
KnowledgeGraphView 把数据传给 GraphRenderer，
GraphRenderer 使用 ECharts graph series 把知识图谱渲染到画布上。
```

---

## 三、两个流程的关系

知识图谱在项目里有两个作用：

1. **可视化展示**

```text
Neo4j -> GraphService -> GraphController -> graphStore -> GraphRenderer -> ECharts
```

2. **参与智能问答 RAG**

```text
用户问题 -> QueryRewriteService -> KnowledgeRetrievalService -> Neo4j 节点/关系证据 -> systemPrompt -> 大模型回答
```

所以可以这样理解：

```text
知识文档提供“文本证据”，
知识图谱提供“节点和关系证据”，
两者一起拼进 systemPrompt，
帮助大模型生成更贴合项目知识库的回答。
```
