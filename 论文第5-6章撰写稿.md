# 第5章 系统测试

本章围绕智能编程学习助手系统的测试方法、测试方案与计划、测试过程及结果分析展开。测试目标是验证系统在真实使用场景下的功能正确性、接口稳定性、模块协同能力与异常可恢复能力，为系统验收提供依据。

## 5.1 测试方法

本系统采用“功能测试 + 接口契约测试 + 服务层单元测试 + 联调测试”相结合的方法，兼顾用户视角与工程实现视角。

（1）功能测试（黑盒测试）  
从学习者用户角度验证核心业务流程是否可用，重点覆盖用户认证、智能问答、知识图谱、学习路径、代码片段、学习分析与报告导出等模块。

（2）接口契约测试  
针对 `/api/auth`、`/api/chat`、`/api/graph`、`/api/path`、`/api/snippets`、`/api/progress`、`/api/documents` 等接口，验证参数校验、鉴权、统一响应结构与异常返回是否符合预期。

（3）服务层测试（白盒测试）  
围绕 `ChatService`、`DocumentService`、`GraphService`、`PathService`、`ProgressService`、`FeedbackService` 等核心服务，验证关键逻辑分支与边界场景，包括缓存命中/未命中、检索回退、节点状态更新与报告生成。

（4）系统联调测试  
在 Docker 多服务环境中验证前端、后端、MySQL、Redis、Neo4j、Elasticsearch 之间的协同能力，确认系统在真实部署场景下稳定运行。

（5）异常与回归测试  
对未登录访问、外部模型鉴权失败、检索服务不可用、图谱查询异常等场景进行验证，确认系统可降级、可提示、可恢复。

### 5.3.2 关键链路与性能现象

结合运行日志与联调过程，系统表现如下：

1. 后端冷启动通常在 8~10 秒内完成，满足本地开发与演示需求。  
2. 智能问答检索链路存在“快慢双态”：常规问题在百毫秒级完成检索，复杂问题在秒级返回。  
3. Elasticsearch 不可用时，系统可回退到本地检索流程，保证问答不中断。  
4. Redis 上下文缓存与问答缓存生效后，同会话连续提问的处理更稳定。  
5. 图谱、路径、文档、片段、进度模块在统一鉴权与统一响应下可稳定协同。

### 5.3.3 缺陷修复与回归结论

测试阶段重点修复了以下问题：

1. `knowledge_chunk` 表缺失导致问答 500：已通过建表与初始化修复。  
2. Elasticsearch 端口冲突导致连接失败：已调整容器与端口配置并验证连通。  
3. 文档/向量维度不一致导致索引写入告警：已增加维度检测与重生成逻辑。  
4. 外部模型调用 401：已完成密钥配置校验与异常提示。  
5. 页面空间冲突（评估面板挤占聊天区）：已优化为按钮折叠展开交互。  
6. 图谱健康度面板占用画布：已支持折叠显示，提升画布可用面积。

回归结果表明：已修复问题未引入明显回归故障，核心功能持续可用。

### 5.3.4 现阶段不足

虽然系统功能完整，但仍有进一步优化空间：

1. 检索质量评价指标（如 Recall@K、MRR）尚未形成自动化报表。  
2. 压力测试规模有限，缺少更高并发下的量化结论。  
3. 图谱规模扩大后，前端布局性能仍需进一步优化。  
4. 推荐权重仍以规则为主，后续可引入学习型排序策略。

## 5.4 本章小结

本章从方法、方案、执行与结果四个层面完成了系统测试论证。测试结果表明：系统已具备较好的功能完备性与工程可用性，能够稳定支撑“智能问答 + 知识图谱 + 学习路径 + 代码片段 + 学习分析”的核心业务链路。与此同时，性能评测深度和检索评估自动化仍是后续优化重点。

# 第6章 总结与展望

## 6.1 工作总结

本文围绕“智能编程学习助手”完成了从需求分析、系统设计、核心实现到测试验证的完整工程实践，主要工作如下：

1. 完成了前后端分离架构设计与实现，构建了可持续迭代的工程基础。  
2. 以学习者用户为核心，完成了认证、问答、图谱、路径、片段、分析、文档七大业务模块。  
3. 构建了 RAG 问答链路，实现“检索增强 + 上下文构造 + 流式生成 + 来源展示 + 用户反馈”闭环。  
4. 引入 Neo4j 图谱能力，实现知识节点关系可视化、依据联动与学习路径增强。  
5. 引入 Redis 缓存机制，提升高频问答与多轮会话场景下的响应稳定性。  
6. 形成 Docker 化部署方案，提升环境一致性和演示复现效率。

从课设目标达成角度看，系统已满足“可运行、可演示、可扩展、可说明”的总体要求。

## 6.2 研究与实现价值

本项目的价值主要体现在三方面：

1. 工程价值：完整覆盖了一个智能学习系统的主链路，具备较强实践参考意义。  
2. 教学价值：将问答、图谱、路径、分析串联为学习闭环，提升学习过程可追踪性。  
3. 技术价值：验证了“知识图谱 + 检索增强 + 大模型”在编程学习场景中的可落地性。

## 6.3 存在不足

当前系统仍存在以下不足：

1. 知识库规模相对有限，对长尾问题覆盖能力仍需增强。  
2. 检索评估以在线统计为主，缺少标准数据集下的离线评测体系。  
3. 问答性能尚未在高并发场景下进行系统压测。  
4. 代码推荐与学习建议的个性化程度仍有提升空间。  
5. 移动端与多端一致性体验仍需进一步优化。

## 6.4 后续展望

面向后续迭代，可按以下方向推进：

1. 检索质量提升：引入更高质量 embedding 模型，完善 Query Rewrite 与重排策略。  
2. 评测体系建设：建立离线评测集，持续输出召回率、排序质量与答复可用率指标。  
3. 性能与稳定性：补齐限流、熔断、监控告警与压测基线。  
4. 图谱持续建设：扩展节点类型与关系类型，提升知识覆盖与可解释性。  
5. 个性化学习：构建更细粒度用户画像，实现动态路径规划与差异化推荐。  
6. 产品化完善：优化交互细节与可视化表达，提升长期学习场景下的用户体验。

## 6.5 本章小结

本章总结了系统建设成果与实际价值，分析了当前阶段的不足，并给出了下一步优化方向。总体而言，本系统已经完成毕业设计阶段目标，并具备继续演进为“数据驱动、可解释、可持续优化”的智能学习平台的基础能力。
